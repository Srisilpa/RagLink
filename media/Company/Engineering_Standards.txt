1. Introduction

Engineering Standards defines the mandatory conventions for writing, reviewing, and shipping
code at Series Tech Ltd. It applies to all Engineering, DevOps, and Architecture teams
contributing to shared repositories, including Project Alpha and internal platform services.

Note: Standards in this document are enforced through automated linting in CI (see Deployment
Guide & QA Testing Guidelines, STL-DEVQA-DEP-008) as well as manual code review.

1.1 Technology Stack Overview

SeriesTech’s primary stack includes Django and Spring Boot for backend services, React and
Angular for frontend applications, PostgreSQL and MySQL for relational data, Redis for caching
and queues, and Docker/Kubernetes for containerized deployment. AI-oriented services use
LangChain, ChromaDB, FAISS, Ollama, and Llama models, alongside OpenAI APIs where approved for
client use.

2. Coding Standards

2.1 General Principles

-   Code must be self-documenting where possible; comments explain “why,” not “what.”
-   Functions should do one thing; a function longer than 50 lines is a candidate for
    refactoring.
-   No hardcoded credentials, API keys, or connection strings — use environment variables or
    the secrets manager (Azure Key Vault / AWS Secrets Manager).

2.2 Language-Specific Guidance

Python (Django, LangChain services): Follow PEP 8. Use type hints for all public functions.
Format with black, lint with ruff.

Java (Spring Boot): Follow Google Java Style Guide. Use Lombok sparingly, only for DTOs, never
for domain entities with business logic.

JavaScript/TypeScript (React, Angular): Prefer TypeScript for all new frontend code. Use ESLint
with the SeriesTech shared config (@seriestech/eslint-config), and Prettier for formatting.

2.3 Example — Acceptable vs. Non-Compliant Code

Non-compliant:

    def proc(x, y, z):
        conn = psycopg2.connect("host=prod-db user=admin password=Secret123")
        ...

Compliant:

    def calculate_leave_balance(employee_id: int, leave_type: str, year: int) -> float:
        conn = get_connection(settings.DATABASE_URL)
        ...

3. Naming Conventions

3.1 Variables and Functions

Use snake_case for Python variables and functions, camelCase for JavaScript/TypeScript and Java
variables/methods, and PascalCase for classes and React components across all languages.

3.2 Repository and Service Naming

Repositories follow the pattern st-<domain>-<service>, for example st-alpha-api or
st-alpha-frontend. Microservices follow <domain>-service, for example auth-service,
ingestion-service, retrieval-service.

3.3 Database Naming

  Object        Convention                       Example
  ------------- -------------------------------- --------------------------
  Table         snake_case, plural               employee_records
  Primary Key   id                               id
  Foreign Key   <referenced_table_singular>_id   employee_id
  Index         idx_<table>_<column>             idx_documents_created_at

4. Git Workflow

4.1 Repository Structure

All source code resides in GitHub Enterprise under the seriestech organization. Each repository
must include a README.md, CONTRIBUTING.md, and a CODEOWNERS file mapping directories to
responsible teams.

4.2 Commit Message Standards

Commits follow Conventional Commits: <type>(<scope>): <description>, where type is one of feat,
fix, docs, refactor, test, or chore.

Example: feat(retrieval-service): add hybrid BM25 + vector reranking

6. Pull Request Process

6.1 PR Requirements

Every PR must link a Jira ticket, include a description of the change and testing performed,
and pass all CI checks (lint, unit tests, security scan) before requesting review.

6.2 RACI Matrix — Pull Request Lifecycle

  Activity               Author   Reviewer   Tech Lead   QA
  ---------------------- -------- ---------- ----------- ----
  Write code and tests   R        I          I           I
  Request review         R        I          I           I
  Review code            I        R          C           I
  Approve merge          I        A          C           I
  Merge to develop       R        I          A           I
  Sign off for release   I        I          C           R

(R = Responsible, A = Accountable, C = Consulted, I = Informed)

6.3 Merge Requirements

At least one approval from a designated CODEOWNER is required; PRs touching authentication,
payment, or client data pipelines require two approvals, including one from the Architecture
team.

7. Code Reviews

7.1 Review Checklist

-   Does the code follow the Coding Standards in Section 2?
-   Are naming conventions from Section 3 respected?
-   Is test coverage adequate (minimum 80% for new modules)?
-   Are there any security concerns, such as unvalidated input or exposed secrets?
-   Is documentation updated per Section 8?

Best Practice: Reviewers should respond to a PR within 1 business day. If a large PR (500+
lines) is submitted, the author should proactively request a pairing session rather than
waiting for asynchronous review.

7.2 Review Etiquette

Feedback should be specific and actionable, referencing line numbers and suggesting
alternatives rather than only flagging problems. Disagreements that cannot be resolved between
author and reviewer should be escalated to the Tech Lead, not left unresolved in PR comments.

8. REST API Development Guidelines

8.1 API Design Principles

All REST APIs follow resource-oriented URL design (/api/v1/documents/{id}), use plural nouns
for collections, and return standard HTTP status codes. Versioning is done via URL path
(/api/v1/, /api/v2/), never via headers alone.

8.2 API Table — Standard Endpoints (Example: Document Service)

  ------------------------------------------------------------------------------------------------
  Method                  Endpoint                 Description             Auth Required
  ----------------------- ------------------------ ----------------------- -----------------------
  GET                     /api/v1/documents        List documents with     Yes (JWT)
                                                   pagination              

  GET                     /api/v1/documents/{id}   Retrieve a single       Yes (JWT)
                                                   document                

  POST                    /api/v1/documents        Ingest a new document   Yes (JWT + role check)
                                                   into the RAG pipeline   

  PUT                     /api/v1/documents/{id}   Update document         Yes (JWT + role check)
                                                   metadata                

  DELETE                  /api/v1/documents/{id}   Remove document and     Yes (JWT + admin role)
                                                   associated vector       
                                                   embeddings              
  ------------------------------------------------------------------------------------------------

8.3 Authentication and Error Handling

All internal APIs require a valid JWT issued by the Auth Service (OAuth 2.0 Client Credentials
or Authorization Code flow, depending on caller type). Error responses follow a standard
envelope:

    {
      "error": {
        "code": "DOCUMENT_NOT_FOUND",
        "message": "Document with id 4821 was not found",
        "trace_id": "a1b2c3d4"
      }
    }

Warning: Never return raw stack traces or database error messages in API responses; log them
internally and return a generic, coded error to the client.

9. Documentation Standards

9.1 Required Documentation per Repository

Each repository must maintain an up-to-date README.md covering setup instructions, an
ARCHITECTURE.md for services with more than one component, and inline docstrings for all public
functions and classes.

9.2 API Documentation

REST APIs must be documented using OpenAPI 3.0 specifications, auto-generated where possible
from Django REST Framework or Spring Boot annotations, and published to the internal Developer
Portal.

9.3 Documentation Review Cadence

Documentation is reviewed each quarter alongside the Architecture team’s system review (see
Enterprise System Architecture & FAQ, STL-ARCH-SYS-009). Stale documentation flagged during
this review must be updated within 2 sprints.

9A. Testing Expectations for Engineering

9A.1 Unit Testing Requirements

Every new module must ship with unit tests achieving at least 80% line coverage, enforced by
the unit-test CI job described in the Deployment Guide & QA Testing Guidelines
(STL-DEVQA-DEP-008), Section 5.1. Test files mirror the source structure, for example
services/retrieval.py is tested by tests/services/test_retrieval.py.

9A.2 Mocking External Dependencies

Calls to external LLM providers (OpenAI APIs) or vector stores (ChromaDB, FAISS) must be mocked
in unit tests using recorded fixtures, never live calls, to keep the test suite deterministic
and fast. Integration tests running against testcontainers-provisioned ChromaDB instances are
the appropriate layer for validating actual retrieval behavior.

9B. Performance and Code Efficiency

9B.1 Performance Guidelines

-   Avoid N+1 database queries; use select_related/prefetch_related in Django ORM code, or
    equivalent JPA fetch joins in Spring Boot.
-   Batch embedding generation calls where possible rather than issuing one API call per chunk,
    to reduce latency and cost for LangChain-based ingestion pipelines.
-   Cache expensive, frequently repeated computations (for example, tenant configuration
    lookups) in Redis with an appropriate TTL.

9B.2 Example — Batched Embedding Call

Non-compliant (one call per chunk):

    for chunk in chunks:
        embedding = embed_client.embed(chunk.text)
        store.add(chunk.id, embedding)

Compliant (batched call):

    texts = [chunk.text for chunk in chunks]
    embeddings = embed_client.embed_batch(texts)
    store.add_many(chunk_ids=[c.id for c in chunks], embeddings=embeddings)

9C. Deprecation and Technical Debt

9C.1 Deprecation Process

Deprecated functions or endpoints must be marked with a @deprecated decorator (Python) or
@Deprecated annotation (Java), referencing a Jira ticket for removal, and must remain
functional for at least one full release cycle (per the bi-weekly release train in
STL-DEVQA-DEP-008, Section 7.1) before removal.

9C.2 Technical Debt Tracking

Technical debt items are logged in Jira under the TECH-DEBT label and reviewed monthly by each
Tech Lead. Items affecting Project Alpha’s RAG pipeline reliability are escalated to the
Architecture Review Board for prioritization against the quarterly roadmap.
